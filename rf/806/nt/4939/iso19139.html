Abstract:

For decades, California fishery reporting and management has been organized and documented using a 10 minute latitude-longitude (10 minute) grid, known colloquially as 'California Trawl Blocks'. In recent years, however, a need has developed for finer (smaller area) grid blocks to record catch and develop habitat maps and models, especially in coastal nearshore waters and for residential and sessile species. This polygon shapefile of 1 minute latitude-longitude (1 minute) blocks was created to nest within the original 10 minute latitude-longitude blocks, thus allowing the transfer of 10 minute historic and current catch data to each of the 100 1 minute cells within. The 1 minute blocks in this shapefile extend inland to include coastal areas, estuaries and bays. This shapefile is a polygon file, not a raster file, therefore, area for each 1 minute cell may be unique.

Purpose:

This coverage provides a systematic polygonal grid that allows the ocean to be divided into 1 minute latitude-longitude administrative blocks for the purpose of management, analysis and defense of California's natural marine resources. Specifically, the California Recreational Fisheries Survey (CRFS) project has adopted these block sequence codes to gather recreational fisheries information statewide.

Process Step:

Metadata imported. I. Acquiring the Extension Through a search on the Web, ET Geowizards extension was discovered. The Extension is an unregistered copy available on the web for free. To obtain the full version Registration and $140.00 is required. II. Drawing the One minute latitude -longitude Grid. 1. Under the Basic Tab, select the "Generate Vector Grid". 2. You must have a shapefile that is added to your project that contains the extent to which you want your Grid Drawn. (There must be no projection defined)!! See Notes Below. Notes: a. The initial extents of the grid are defined by the extents of the selected layer or current extents of the data frame. The extents can be changed manually during the dialog. b. The cell size will be in the units of the input layer or data frame. If the input is Unprojected (Geographic "Projection") there is an oprion to input values in Degrees - Minutes - Seconds c. If the input is Unprojected only values between X = -180 To 180 and Y = -90 To 90 will be accepted d. If the Input is Projected and the Output is in different projection, the user should make sure that the Grid extents are valid e. In order to avoid incorrect inputs, the size of the grid is limited to 4,000,000 cells (2000 by 2000) f. A ET_Index field will be added to the attribute table. g. Grid is read Bottom -to-top Left-to-right. (Row-Column) 3. Select source for initial Grid extent: Caltrawl3_Nad27 (unprojected Shapefile) 4. Specify Output Table: 1NM_Caltrawl3_Nad27_Vector.shp : Save 5. Select spatial reference source layer: Caltrawl3_Nad27 6. Select GRID type: Polygon 7. Grid extents: DEGREE 8. Cell size: DEGREE : DMS = 1 minute for X and Y (See Diagram below) 9. Click Finish III. Indexing the new Shapefile The Resulting Grid that was drawn was superimposed using the Latitude (North-South) and longitude (East-West) boundaries of the inputted shapefile(Caltrawl). The resulting Grid had to be "clipped" to the Caltrawl file, however in order to include an index that was systematic, the Grid had to include blocks that were superimposed onto land. The process used to create the 1 minute latitude -longitude grid was repeated to create a 10 nautical mile grid over land. In this way a new index could be systematically read Left-to-right, top-to-down, 1-to-100 for an entire 10 Nautical mile Block, using the Caltrawl Block_10 ID as a prefix. Some 10 mile blocks had to be labeled if they were going to contain the same Block value, this was accomplished by adding a 1, 2, 3, etc at the end of the 3 digit Block10_ID code. note: Having land based blocks may prove useful in the future if a weighted probability scoring model will be used for a estimating available rocky Habitat. IV. Dissolving shapefiles in ArcGIS . We wanted to reference all the one minute latitude -longitude blocks to the corresponding 10 nautical mile blocks they were nested in, using the Block10_ID as a prefix for the new index. This can be accomplished by performing a spatial Join so that the attributes of one shapefile (10nm) can be appended to another shapefile (1nm) if it falls completely within that 10 mile block. Both vector grid shapefiles however did not line up exactly and was not drawn perfectly though both grids were drawn on Latitude and Longitude lines. Typically by dissolving shapefiles on a common attribute, polygons can be redrawn, removing slivers and non block polygons. The unregistered copy of ET Geowizards does not support shapefiles with more than 100 records. The Geoprocessing wizard in Arcgis 8.2 will also dissolve polygons however with 72,282 records, the program would hangup and "not respond". V. Using XTools Xtools is an extension that allows for additional geoprocessing functions. The Calculate area Function allows for the calculation of area, perimeter, length, acres and hectares of polygons. Sorting records in ascending order on Area, I was able to identify 1,282 records that were noticeably smaller than the other blocks. By counting the existing Caltrawl 10 mile blocks, I calculated that 71,600 1 mile blocks should be represented. The resulting Shapefile has 71,500 records (100 land-based blocks in San Diego that were inadvertently truncated during the vector drawing process). The shape to Centroid tool in xtools will replace a polygon with a point in the center of all existing polygons in a shapefile. The 1 minute latitude -longitude polygon blocks were reduced to points and then buffered to .15 nautical miles using the Buffer wizard under the tools menu. VI. Spatial Joins When performing spatial joins, the attributes of the two shapefiles needing to be appended must both be polygons. The 1 minute latitude -longitude blocks must be completely inside the 10 nautical mile block in order for the 1 nm shapefile to take on the 10 nm attributes, otherwise the attributes for that cell will be blank. In this case boundaries overlap so there are thousands of records missing attribute information, too many to manually assign. By converting the 1 nm polygons to centroids and then buffering them so that they would be a quarter of their original size, all corresponding 1nm buffered centroids were assigned their appropriate Caltrawl10_ID Attribute. VII. Joins based on a common attribute The ET_index originally created by ET Geo Wizards was used to "re-append" these attributes back to the original 1 Nautcal mile shapefile. Unfortunately trying to do this join in ArcGIS on 71,500 records caused the software to hangup. Instead, the 1nm polygon shapefile was exported into a Geodatabase and the 1nm buffered centroids attribute table was exported as a *.dbf file. VIII.Using ACCESS and EXCEL to Re-index and Append the new index Whether it was due to the number of records in the dataset, software glitches in ARCGIS 8.2 and/or hardware compatability issues, appending a new index to the 1 nm shapefile required creativity and resourcefulness. Microsoft EXCEL can only accomodate 65,000 records at a time. The DBF file containing 71,500 records was brought into ACCESS, a primary key added and the dataset exported to EXCEL as two datasets. The ET-Index field was then separated using the '- ' character in the "text-to-columns" function; The first set of numbers were labeled RowET_Index and the last set of numbers labeled Column. The entire dataset was then resorted using Block10_ID, RowET_Index and Column so that records read in sequential order left-to-right, top-to-bottom for each Block. A new field was created called New ID, and values 1-100 appended to 10x10nm blocks, 1-600 for all 20x30 nm blocks, 1-900 for 30x30 nm blocks, 1-50 for the two blocks in San Diego that were truncated. These files were saved as text files and then re-imported in to ACCESS, appending both datasets back into one table. The table was imported back into ARCGIS, a new field created and populated using the field calculator to append Block10_ID, '-', and the new ID together. The table was then exported into the Geodatabase, and a make table query created that replaced the existing 1 nm attribute table.
